# UPDATE — Sistem Variant, Costing (WAC), Konversi, Report & Opname

> Dokumentasi perubahan besar pada sistem inventory.
> Inti perubahan: **sumber kebenaran stok pindah sepenuhnya ke `product_variants`**,
> dan **harga modal (buy price) sekarang dihitung otomatis** memakai metode
> Weighted Average Cost (WAC) di setiap kejadian yang memindahkan barang.

---

## 1. Ringkasan Non-Teknis

Sebelumnya stok dicatat di dua tempat (product dan variant) dan harga modal
diisi manual. Sekarang alurnya jadi satu jalur yang utuh, meniru alur fisik
barang di lapangan:

```
KULAKAN (per kg)
   │  masuk ke variant "default", harga modal semua variant
   │  dihitung ulang otomatis (rata-rata tertimbang)
   ▼
PRODUCT CONVERTER (opsional)
   │  ikan utuh → fillet, boleh menyusut (5 kg → 3 kg),
   │  biaya buangan otomatis terserap ke harga modal hasil jadi
   ▼
VARIANT CONVERTER (repack)
   │  curah default → cup 500gr / 250gr / 100gr,
   │  stok berpindah sesuai pengali (conversion factor)
   ▼
PENJUALAN (per variant)
   │  stok keluar dari variant yang dijual,
   │  profit dihitung dari snapshot harga saat order
   ▼
STOCK OPNAME
      selisih fisik vs sistem tercatat per item,
      kerugian susut terlihat dalam rupiah
```

Semua kejadian di atas **tercatat di ledger masing-masing**, jadi setiap
perubahan stok dan harga selalu bisa dilacak: kapan, berapa, oleh siapa,
dan nilainya berapa rupiah.

---

## 2. Konsep Kunci

### 2.1 Variant "default"

- Setiap product **wajib** punya satu variant bernama `default` —
  representasi barang curah/utuh product itu sendiri (satuan kg).
- Kulakan selalu masuk ke default. Konversi selalu lewat default.
- `products.stock` dan `products.buy_price` sekarang hanya **mirror**
  dari variant default (disinkronkan otomatis oleh RPC), bukan sumber data.

### 2.2 Conversion Factor

Kolom baru `product_variants.conversion_factor numeric(10,4)` =
**berapa kg yang dikonsumsi per 1 unit variant**:

| Variant     | Factor |
|-------------|--------|
| default     | 1      |
| cup 500 gr  | 0.5    |
| cup 250 gr  | 0.25   |
| cup 100 gr  | 0.1    |

Dipakai untuk: repack, agregasi total kg, dan distribusi harga modal.

### 2.3 Weighted Average Cost (WAC)

Setiap barang masuk (kulakan / konversi product), harga modal dihitung ulang:

```
              Σ(stock_i × buy_price_i) + nilai_barang_masuk
avg (per kg) = ──────────────────────────────────────────────
              Σ(stock_i × conversion_factor_i) + qty_masuk_kg

buy_price_i (baru) = avg × conversion_factor_i     ← semua variant
```

**Contoh kulakan:** default 2 kg @21.000 + cup500 4 pcs @11.000
→ aset 86.000, setara 4 kg → avg lama 21.500/kg.
Kulakan 6 kg @20.000 (120.000) → aset 206.000 / 10 kg = **20.600/kg**.
Default → 20.600, cup500 → 10.300, cup250 → 5.150.

**Contoh konversi + penyusutan:** 5 kg patin (buy 20.600) → fillet,
jadi hanya 3 kg. Nilai pindah = 103.000 **seluruhnya** dibebankan ke 3 kg
→ cost fillet naik. Buangan (kepala/tulang) otomatis jadi beban produk jadi
— ini costing yang benar.

> Konsekuensi: buy_price variant **bukan lagi angka manual** — tertimpa
> setiap ada kulakan/konversi. Snapshot harga di `order_items` menjaga
> profit historis tetap akurat.

## 3. Perubahan Database

### 3.1 Kolom baru
```sql
alter table product_variants
add column conversion_factor numeric(10,4) not null default 1;
```

### 3.2 Table ledger baru (3)

| Table | Mencatat | Kolom penting |
|---|---|---|
| `variant_conversion_ledger` | Repack default ↔ variant | qty, factor (snapshot), base_qty |
| `product_conversion_ledger` | Konversi A → B + biaya | qty_from/to, unit_cost, total_value, new_avg_cost |
| `stock_opname_ledger` | Selisih opname per item | stock_before/after, difference, unit_cost, value_diff, session_id |

Semua ledger men-snapshot harga/factor **saat kejadian**, jadi riwayat
tetap valid walau master berubah kemudian.

### 3.3 RPC baru / diubah

| RPC | Status | Fungsi |
|---|---|---|
| `product_purchase_transaction` | **diubah** (input tetap) | Stok masuk ke variant default + WAC repricing semua variant + sync mirror products |
| `convert_product_variant` | baru | Repack dua arah (pack/unpack), validasi stok, lock, tulis ledger |
| `convert_product` | baru | A→B via default, penyusutan (qty_from ≠ qty_to), cost absorption + WAC di B |
| `update_product_with_default_variant` | baru | Edit product & variant default sinkron dalam 1 transaksi |
| `stock_opname` | baru | Batch set stok fisik, catat selisih per item, skip selisih 0 |
| `report_product_sales` | baru | Per product: qty terjual (kg), revenue, cost, profit, kulakan |
| `report_sales_summary` | baru | Ringkasan: revenue, ongkir, COGS, gross profit, expense, purchase |

`apply_order` & `update_order_status` **tidak diubah** — sudah bekerja di
level variant sejak awal.

### 3.4 Pola teknis yang dipakai konsisten

- **Locking**: `SELECT ... FOR UPDATE` sebelum baca-hitung-tulis.
  Konversi product me-lock kedua product **urut `product_id`** (anti-deadlock).
- **Fallback WAC**: kalau total setara-kg ≤ 0 atau aset negatif (ada stock
  minus), pakai harga barang masuk hari itu — tidak error, tidak memblokir.
- **Range report half-open** `[start, end)`: hindari bug `23:59:59`.
- **Qty penjualan diagregasi ke kg** via factor; item lama tanpa variant
  dianggap factor 1.
- **Definisi penjualan di report**: order status `paid`/`picked up`/
  `delivered`, basis `order_date`.

## 7. Known Caveats / Next

- Penjualan **tidak memvalidasi stok** (bisa minus) — by design, tapi sadari.
- Kulakan **belum bisa di-void** — tricky karena WAC sudah menimpa harga.
- Buy price manual variant tertimpa WAC — kalau butuh biaya packaging per
  cup, tambah kolom `packing_cost` (belum dibuat).
- **RLS belum diaktifkan** — prioritas keamanan berikutnya.
- Opname sebaiknya saat tidak ada transaksi berjalan (masalah proses,
  bukan kode — lock sudah menjaga integritas data).
