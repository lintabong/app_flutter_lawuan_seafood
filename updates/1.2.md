
# Update — Edit Order Feature

Ringkasan perubahan untuk menambahkan kemampuan **edit order** (khusus status `pending`), plus perbaikan filter tanggal dan perilaku item duplikat.

## Scope edit

Order **hanya bisa diedit saat status `pending`** (dijaga ganda: di UI tombol Edit disembunyikan kalau bukan pending, dan di RPC lewat guard exception).

Yang bisa diedit: order items (variant, qty, sell price), delivery type, delivery fee, dan tanggal order.

Yang **tidak** bisa diedit: customer, status, dan cash account (disembunyikan).

Alasan aman tanpa reversal: order `pending` belum pernah menyentuh stok, transaksi, maupun cash ledger (efek finansial di `apply_order` hanya jalan saat status `paid`/`picked up`/`delivered`). Jadi edit cukup: validasi ulang item, recompute total, update header, hapus `order_items` lama lalu insert ulang.

## Detail per file

### 1. `edit_order.sql` (RPC baru)

Signature: `edit_order(p_order_id, p_items, p_delivery_price, p_delivery_type, p_order_date)`.

Guard: kalau `status <> 'pending'` → raise exception. Validasi item sama seperti `apply_order` (qty > 0, variant milik product). Recompute `total_amount`, update kolom `orders`, lalu delete-reinsert `order_items`. Nol logika stok/kas/transaksi.

Jalankan file ini di SQL editor Supabase.

### 2. `edit_order_page.dart` (halaman baru)

Tiga section: tanggal, delivery (toggle + fee), dan items editor. Customer tampil di header sebagai read-only. Item di-seed dari `order['order_items']` lalu di-resolve ke object product + variant lengkap, dengan fallback jika produk sudah non-aktif.

### 3. `order_pickers.dart` (file baru)

`ProductPickerSheet` dan `CustomerPickerSheet` diangkat jadi public (sebelumnya private di dalam `create_order_page.dart`), supaya dipakai bersama oleh create & edit.

### 4. `order_card.dart` (diubah)

Ditambah tombol **Edit** (aksen ungu) di bawah tombol Invoice. Tombol hanya muncul saat `order['status'] == 'pending'` (variabel `canEdit`). Ada parameter baru `onEdit`.

### 5. `order_page.dart` (diubah)

Menyambungkan `onEdit` → `_openEdit()` yang push `EditOrderPage` dan reload list kalau kembali dengan hasil. Filter tanggal: `lastDate` diubah dari `DateTime.now()` menjadi `DateTime(2100)` supaya bisa memilih tanggal beberapa tahun ke depan (konsisten dengan create order).

### 6. `create_order_page.dart` (diubah)

`_addProductSheet` sekarang **selalu menambah baris baru** — variant yang sama boleh muncul sebagai beberapa baris terpisah (mis. kakap fillet 0.4 kg dan 0.7 kg). Sebelumnya logika lama menggabung qty jika variant sudah ada. Dua picker private dihapus, diganti import dari `order_pickers.dart`.

### 7. `supabase_service.dart` (method baru)

Tambah `editOrder({orderId, items, deliveryPrice, deliveryType, orderDate})` yang memanggil RPC `edit_order`. Mengirim tanggal sebagai UTC ISO string.

## Catatan penting

- **Path import** — asumsi `edit_order_page.dart` dan `order_pickers.dart` berada di folder `pages/` (sejajar `order_page.dart` & `create_order_page.dart`). Sesuaikan `import` jika struktur folder berbeda.
- **`getOrders` select** — pastikan join `order_items` mengembalikan `product_id`, `product_variant_id`, serta join `products` (`id, name`) dan `product_variants` (`id, name, sell_price, buy_price, unit`). Ini yang membuat seeding item di edit page akurat.

## Known issue / fix

**Error `column reference "order_id" is ambiguous` (42702)** saat memanggil RPC.

Penyebab: nama kolom output `order_id` (dari `returns table(order_id ...)`) bentrok dengan kolom tabel saat query. Solusi yang direkomendasikan:

1. Rename kolom OUT jadi `out_order_id` / `out_total_amount` di `returns table(...)`.
2. Qualify DELETE pakai alias: `delete from order_items oi where oi.order_id = p_order_id;`

Dua-duanya menghilangkan semua kemungkinan ambiguity. Kalau kolom OUT di-rename, sesuaikan parsing di Dart (atau abaikan karena edit page memakai `orderId` lokal, bukan dari hasil RPC).
